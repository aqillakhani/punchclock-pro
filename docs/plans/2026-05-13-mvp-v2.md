- [ ] **A4. Hard-cap enforcement in `punchIn`** — `packages/api/src/services/time-tracking.service.ts`

  - Add `AppError.capExceeded` helper.

  - 4 new unit tests: daily cap blocks; weekly cap blocks; warn mode returns warning; exempt window skips.

## Phase B · Self-service + compliance (Day 3–4)

Goal: workers can do their own time-off & shift swaps; meal-break enforcement; selfie on punch.

- [ ] **B1. Worker self-service routes** — `packages/api/src/routes/me.ts` (new)
  - `GET  /api/v1/me/timesheet?from=&to=` (one-user version of admin/timesheets).
  - `GET  /api/v1/me/schedule?from=&to=`.
  - `POST /api/v1/me/time-off` + `GET /api/v1/me/time-off`.
  - `POST /api/v1/me/shift-trade` + `POST /api/v1/me/shift-trade/:id/accept`.
  - Tests: each happy path + auth/role denials.

- [ ] **B2. Manager approval endpoints** — `packages/api/src/routes/admin.ts`
  - `POST /api/v1/admin/time-off/:id/decision` (approve|reject + comment).
  - `POST /api/v1/admin/shift-trade/:id/decision` (final manager confirmation).
  - On approval of time-off, create a placeholder shift block to prevent double-booking.
  - Tests.

- [ ] **B3. UI: My Timesheet + My Schedule** — `packages/web/src/app/dashboard/my-timesheet/page.tsx`, `…/my-schedule/page.tsx`
  - Single-row weekly hours for current user.
  - Schedule shows only their own shifts; cannot edit.

- [ ] **B4. UI: Time off** — `packages/web/src/app/dashboard/time-off/page.tsx`
  - Worker view: "Request time off" form + my requests list with status.
  - Manager+ view: same page renders a pending-approvals queue with approve/reject buttons. Conditional via `<Gate>`.

- [ ] **B5. UI: Shift trades** — `packages/web/src/app/dashboard/trades/page.tsx`
  - Same pattern: workers see open trades + post their own; managers see pending approvals.

- [ ] **B6. Meal-break enforcement** — `packages/api/src/services/break.service.ts`
  - On punch-out, if shift > 6h and no `break` row exists, surface a warning in the response (`missing_meal_break: true`).
  - If org `worksite='onshore'` and state-of-record is California (we don't have state yet; use `organizations.timezone` heuristic — if `America/Los_Angeles`, enforce CA rules), require explicit 30-min lunch break for shifts ≥ 5h.
  - 2 tests.

- [ ] **B7. Punch verification — owner-configurable** (replaces the original "selfie" task; all methods off by default)
  - **Settings UI:** multi-select toggle for `punch_verification_methods` — any subset of `selfie` / `pin` / `ip` / `device`. Geofence stays a separate (existing) toggle.
  - **PIN flow:**
    - On first login after `pin` is enabled, force a "Set your PIN" screen (4–6 digits, confirmed twice). Store via `users.pin_hash` (bcrypt; column already exists from migration 001).
    - On punch, if `pin` is in `punch_verification_methods`, the mobile + web Clock screens show a PIN keypad. POST to `/punch-in` includes `pin`; API verifies via bcrypt compare.
    - Honest UX note: render a tooltip on the toggle: "PIN sharing is the most common buddy-punching vector. Combine with device-pinning for real protection."
    - Owner can reset a worker's PIN from Team page (`POST /admin/users/:id/reset-pin`).
  - **IP flow:**
    - Settings UI: textarea of CIDR ranges (`73.42.18.0/24, …`) → stored in `organizations.allowed_punch_cidrs`.
    - API extracts `req.ip` via `app.set('trust proxy', true)` + Express's `req.ip`. Check membership against CIDRs using `ip-range-check` (tiny dep) or hand-rolled.
    - Block punch with `AppError.ipRestricted({ clientIp })` if not in range.
    - Offshore (`worksite='offshore'`) auto-exempt.
  - **Selfie flow (kept, but off by default):**
    - `packages/mobile/src/services/punch.service.ts`: if `selfie` is in methods, prompt camera before sending.
    - Upload to API as multipart; store filename on `time_entry_events.event_data.photoUrl`.
    - API endpoint: `POST /api/v1/time-tracking/punch-in/photo` (multer + filesystem in `data/photos/<org>/<entry>.jpg` for now; S3 later).
    - On the manager Timesheets row, render a small thumbnail link.
  - **Device pinning:** on first punch from a device, the API records `device_info.deviceId` in a new `user_devices` table. Subsequent punches require a known device unless owner approves a new one (push notification to owner).
  - 6 tests covering each method × success/fail.

## Phase C · Accounting + dashboard (Day 5)

Goal: QuickBooks export works; labor cost visible on the dashboard; predictive-scheduling notice.

- [ ] **C1. QuickBooks IIF export** — `packages/api/src/services/payroll-export.service.ts` (new)
  - Take `from`, `to`, `jurisdiction` → produce IIF lines for each W-2 user (full journal entry) + each 1099 user (contractor expense + AP credit).
  - Use org's `qb_chart_of_accounts` mapping; default account names if unset.
  - Endpoint `GET /api/v1/admin/exports/payroll.iif?from=&to=` streams text response with `Content-Disposition: attachment`.
  - Snapshot test against a hand-verified expected output for the seeded 14-day range.

- [ ] **C2. QBO JSON export** — same service, JSON Journal Entry shape per QBO API docs. Same endpoint suffix `.qbo.json`.

- [ ] **C3. Cost-of-labor endpoint** — `GET /api/v1/admin/cost-of-labor?from=&to=`
  - Returns `{ scheduled: $X, actual: $Y, budget: $Z }`.
  - Scheduled = sum(`shifts.duration_minutes/60 * users.pay_rate`).
  - Actual = sum(timesheet est. pay).
  - Budget = `organizations.weekly_labor_budget` × #weeks in range.
  - Tests.

- [ ] **C4. Overview Labor Cost card** — `packages/web/src/app/dashboard/page.tsx`
  - Owner-only via `<Gate>`. Red if actual > budget; green otherwise.

- [ ] **C5. Predictive scheduling: 14-day lock (opt-in)** — `packages/api/src/routes/scheduling.ts`
  - **Gated by `organizations.feature_predictive_scheduling`.** Off by default.
  - On `POST /shifts` and `DELETE /shifts/:id`, if feature is on and the shift's `scheduled_date` is within the next 14 days, return 409 unless `?force=true` query param + audit log entry.
  - Settings UI: toggle + a small explainer ("Required for retailers in Oregon, San Francisco, Berkeley, LA, NYC, Philadelphia, Chicago, Seattle. See docs.").

- [ ] **C6. Tests + lint**

**Acceptance gate for Phase C:** owner clicks new Export buttons on Reports → downloads `.iif` and `.qbo.json`. Both files import cleanly into a real QB Desktop / QBO test environment (manual check). Overview shows "Labor cost: $1,820 / $1,500 budget" in red when over.

## Phase D · C-store specifics + polish (Day 6)

Goal: cash drawer, documents, audit log, multi-store hooks. Polish for demo recording.

- [ ] **D1. Cash drawer count UI (opt-in)** — `packages/web/src/app/dashboard/clock/page.tsx`
  - **Gated by `organizations.feature_cash_drawer`.** Off by default. Owner enables in Settings if the store wants register-handoff tracking.
  - When clocking IN (in-store onshore only, and feature on): modal prompting "Starting drawer count $___".
  - When clocking OUT: "Ending drawer count $___".
  - Variance flagged on the manager Timesheets row if > $5.
  - Backend endpoint added in design §6.

- [ ] **D2. Employee documents** — `/dashboard/documents` route
  - Worker: upload own I-9 / W-4 / certifications + see expiration alerts.
  - Manager: read all; can mark "verified".
  - Dashboard widget on Overview: "3 documents expiring in 30 days" (owner-only).

- [ ] **D3. Audit log viewer** — `/dashboard/audit-log` (owner-only)
  - Filterable table (date range, actor, action). Audit log table exists; just need the UI + endpoint.

- [ ] **D4. Multi-store stubs**
  - Settings: "Add another store" button creates a second geofence pair + a second `org_locations` row (new minor table).
  - No UI for switching stores yet; defer until owner asks. Schema-ready.

- [ ] **D4a. Kiosk QR clock-in (opt-in)** — `packages/web/src/app/kiosk/page.tsx` (new public route, no auth)
  - **Gated by `organizations.feature_kiosk_qr`.** Off by default.
  - When on: owner prints a QR per employee from the Team page (each QR encodes `https://<host>/kiosk?token=<short-lived-jwt-per-user>`).
  - QR taped at the register. Worker scans with their phone camera → opens kiosk page → 1-tap Punch In/Out.
  - Kiosk page bypasses normal login; the QR token IS the auth (24h validity, one user, store-bound).
  - If `pin` is also enabled in `punch_verification_methods`, kiosk also prompts for PIN — that's the real protection (QR + PIN + IP all enforced).

- [ ] **D4b. Schedule conflict warnings** — `packages/api/src/routes/scheduling.ts` + Schedule modal UI
  - On `POST /shifts`, run pre-flight checks:
    - User already has a shift on that date that overlaps the time range → return 409 with `conflict: 'overlap'`.
    - User's total scheduled hours for the week (including this) > `max_weekly_minutes` → return 409 with `conflict: 'weekly_cap'`.
    - Less than 10h gap between this shift's start and the user's previous shift's end → return 409 with `conflict: 'rest_period'`.
  - UI: the modal renders an inline warning chip *before* submit when fields cause a conflict (live-checked via debounced GET).
  - Manager can `?force=true` to override + audit log.

- [ ] **D4c. Multi-currency display for offshore workers**
  - When a user's `pay_currency != 'USD'`, all pay figures shown to them (My Timesheet, profile) render in both: `$24.00 USD ≈ ₱1,360 PHP`.
  - Conversion rate stored in `organizations.fx_rates JSONB DEFAULT '{}'::jsonb` (manually maintained by owner; one-time entry in Settings). No external FX API call.
  - Manager + owner reports stay in USD.

- [ ] **D4d. Coverage map on Schedule** — `packages/web/src/app/dashboard/schedule/page.tsx`
  - Below the grid, a horizontal 24h × 7-day heatmap showing how many people are scheduled in each hour-slot.
  - Red cells = 0 coverage (gap), amber = 1, green = ≥ 2.
  - Pure derivation from already-loaded shift data; no new endpoints.

- [ ] **D4e. Schedule templates / "Copy last week"** — Schedule page header
  - Two buttons: **Copy last week → this week** (bulk-creates shifts matching last week's pattern) and **Save as template** (stores into `shift_templates`, table already exists from migration 001).
  - Apply-template UI: pick template, pick week-of, click Apply → bulk POST.

- [ ] **D4f. Sign-in-as-worker preview** — owner-only, `/dashboard/preview-as`
  - Owner picks an employee from a dropdown → app re-renders the entire dashboard with that user's role + permissions, banner across the top "Previewing as Alex Rivera — exit preview". Clicks **Exit** → back to owner view.
  - Implementation: a `previewAsUserId` query param + a server-side check (owner only) that overrides `me.role` for the request lifetime. No real session swap; the owner's real JWT stays in place.
  - Why: lets the owner self-train and verify the RBAC matrix is configured correctly without making the owner remember credentials for every test account.

- [ ] **D4g. DB export bundle** — `packages/api/src/db/export.ts` + `pnpm db:export`
  - One command: dumps the entire DB to a JSON file (`backup-YYYY-MM-DD.json`). Restorable via `pnpm db:import`.
  - Useful for backups + cloud migration. Zero ongoing cost.

- [ ] **D5. Push notifications (Expo)** — `packages/mobile/src/services/notifications.service.ts`
  - On login, register the device's Expo push token via `POST /api/v1/me/push-token`.
  - API broadcasts on: schedule change for me, time-off decision, shift trade interest, OT warning.
  - Notification permission prompt at first login.

- [ ] **D6. Docs refresh**
  - `docs/demo-runbook.md` — add the new tabs to the walk-through.
  - `docs/demo-recording-script.md` — extend with worker-view scene + accounting scene.
  - `docs/permissions.md` (new) — public-facing copy of the RBAC matrix.

- [ ] **D7. Final QA pass + commit + push**
  - Playwright walk: login as each role; verify nav matches matrix; verify cap-block fires; verify CSV/IIF download; verify mobile selfie attaches; verify push notification arrives.
